# Purpose

Analyze staged/unstaged changes and create atomic commits by following the `Instructions` and `Workflow`. Whether the result is pushed to the remote is controlled by `PUSH_TO_REMOTE`.

## Variables

PUSH_TO_REMOTE: {{PUSH}}

## Instructions

- Group related changes into separate commits (1 logical change = 1 commit).
- The unit of staging is the whole file: group files into logical commits, never split one file's changes across commits — no `git add -p`/`--patch`, no hunk or line-level staging.
- If a single file mixes concerns, commit it whole under the dominant concern and mention the secondary change in the body.
- Follow conventional commits:

```
<type>(<scope>): <subject>

<body>

<footer>
```

Example:

```
feat(auth): add two-factor authentication

- Implement TOTP generation and validation
- Add QR code generation for authenticator apps
- Update user model with 2FA fields

Closes #123
```

- Types: feat, fix, refactor, style, docs, test, chore
- Keep subject under 50 chars, imperative mood.
- NO "Generated by Claude", "Co-Authored-By", or AI attribution.
- If no changes exist, say so and stop.

## Workflow

1. Run `git status` and `git diff --staged` (if empty, check `git diff`).
2. Identify distinct logical changes.
3. For each change: stage relevant files → commit with clear message.
4. If `PUSH_TO_REMOTE` is true, push the current branch to the remote (set upstream with `-u` if missing).
5. Show final `git log --oneline -n 5` when done.

## Report

- List the created commits (hash + subject).
- If pushed, confirm the branch and remote.
- If anything was intentionally left uncommitted, list it briefly.
