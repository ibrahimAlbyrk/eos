# Eos Design Report — Worktree Awareness & First-Class Test/Preview

> Produced by a five-specialist agent panel (worktree-infra, dx-product, agent-prompts, test-infra, safety) over a research → propose → debate → synthesize pipeline. 2026-06-07.
>
> **Decisions (2026-06-07, maintainer):** P0+P1 ship together as one build — no phased releases. Per-boot UI token is the trust gate (no human-approval row). Hydration as recommended (node_modules clonefile default-on at spawn, `.env*` opt-in). Verdict chip is advisory. `isolation:'none'` not added; the dead composer worktree chip gets deleted. PreviewRunner, CheckRunner, `report_verification` MCP tool — dropped. **Revised later same day:** the integration gesture is **unstaged Try** — a patch-based apply of the merged result into the user's checkout with Keep/Discard, redesigned to avoid `merge --no-commit`'s stuck-MERGE_HEAD failure (the panel's original objection). The daemon `--no-ff` merge-back commit is dropped. Resolutions in §9.

## 1. Executive summary

Orchestrator-dispatched workers always land in isolated git worktrees (`<repo>/.eos/worktrees/eos-*`), but nothing tells them so — no env var, no prompt text, and the `spawn_worker` tool description actively claims they run "in your project directory." Workers therefore tell the user to "run it in your app," which is wrong by construction: their changes are invisible to the user's checkout until merged, and Eos offers no merge, run, or verify affordance — the only exits are agent-run commit/push/PR, and DELETE silently destroys worktree + branch ~2s after kill. The five-specialist panel converged on a layered answer: **(P0)** an awareness pack — spawn-time env vars, a spawner-synthesized per-worker system prompt carrying literal branch/dir facts plus behavioral rules, corrected tool descriptions, and a compliance-independent daemon enrichment of every worker report; **(P1)** make the worktree runnable (APFS-clonefile hydration of `node_modules`), give the user a one-click `/verify` action executed in the agent's own gateway-governed PTY with an honest verdict chip (default "Unverified"), and a deterministic, precondition-gated, undoable daemon merge-back (`git merge-tree` preview → `merge --no-ff` with auto-abort) guarded by a per-boot UI-origin token, plus deletion safety (kill confirm, `eos/trash` tombstone tags, rendering the currently invisible preserved-worktree event); **(P2)** the deferred-by-decision items — a managed dev-server PreviewRunner, a daemon CheckRunner, and dx-product's ephemeral "Try in my checkout," each with named preconditions. Two designs were killed in debate: worktree-scanned "named targets" (agent-writable = gateway bypass) and ff-only daemon merges (routes the common case to the least-constrained executor).

## 2. Current architecture (facts)

**Worktree lifecycle.** Worktrees live inside the user's repo at `<repoRoot>/.eos/worktrees/<branch>`, sharing the main checkout's `.git` object store and refs — `eos-*` branches appear in the user's `git branch` immediately; file changes do not. The branch is generated daemon-side at spawn (`eos-<name>-<workerId>-<clock36>`, `core/src/use-cases/SpawnWorker.ts:84-88`); the **worker process** then creates the worktree itself at boot (`spawner/worktree.ts:35-59`, `git worktree add -b`, always forked from current HEAD — no base-ref parameter exists anywhere). The daemon learns the realpath'd dir asynchronously via the `lifecycle claude_spawning` event (`spawner/worker.ts:78-84` → `core/src/use-cases/ProcessWorkerEvent.ts` → `workers.worktree_dir`, migration 019). Three deletion paths: worker self-teardown at exit removes a clean worktree but preserves a dirty one (emitting a `worktree {phase:preserved, status, diffStat}` event that **nothing consumes** — no `ProcessWorkerEvent` handler, no UI renderer); `DELETE /workers/:id` force-removes worktree **and** `branch -D` after a 2s kill grace regardless of uncommitted/unmerged work (`core/src/use-cases/KillWorker.ts:86-89` → `infra/src/git/ChildProcessWorktreeManager.ts:41-62`), with no UI confirmation (`manager/web/src/views/code/popovers/AgentContextMenu.jsx`); daemon-startup `PruneOrphanWorktrees` removes any managed `eos-*` worktree without a live DB row (`manager/container.ts:102`). Nothing syncs a worktree back to the main checkout today. `--keep-worktree` exists in `spawner/options.ts` but no daemon path emits it.

**Spawn flow.** Three spawn surfaces; only one creates worktrees: the orchestrator MCP `spawn_worker` tool hardcodes `worktreeFrom = session.cwd` whenever the orchestrator's cwd is a git repo, with no opt-out and no cwd/branch/isolation parameter (`manager/orchestrator-mcp/tools/spawn_worker.ts:35`). `POST /orchestrators` has no `worktreeFrom` in its schema — orchestrators always run in the user's real checkout. The web UI never sends `worktreeFrom`; its composer "worktree" chip is dead (`manager/web/src/state/composer.jsx:8` toggled by `ComposerConfigRow.jsx:63`, read by nothing). Worktree rows have `cwd=NULL`; consumers must use the `gitDirOf` precedence `worktree_dir ?? cwd ?? worktree_from` (`manager/routes/workers.ts:40-42`). The claude PTY env carries `EOS_SPAWNED/WORKER_ID/DAEMON_URL` plus inherited `EOS_REPO_ROOT` (which means the **Eos** repo, `manager/container.ts:226`) — nothing identifies worktree-ness, branch, or project root.

**Change visibility.** Three read-only endpoints (`GET /workers/:id/{diff,changes,changes/file}`, `manager/routes/workers.ts:414-448`) all diff **against HEAD only** — once an agent commits in its worktree, the entire UI shows clean; no branch-vs-base diff exists. The Changes panel (`manager/web/src/views/code/messages/DiffViewer.jsx`) is a lazy per-file accordion with SSE-debounced refresh. Git action buttons (`POST /workers/:id/action` → `manager/services/worker-actions.ts` → `manager/prompts/{commit,create-pr}.md`) paste prompt templates into the agent's own PTY (chat shows `/commit` via displayText). `ComposerDiffRow.jsx:124,154` computes branch/ahead/stash chips and the manual-PR URL from `cwd ?? worktree_from` — the **user's** repo for worktree workers, while the diff badge reads the worktree. The web git agent is spawned with `selected.cwd ?? selected.worktree_from` (`Composer.jsx:311-313`) — always the main checkout, never told which `eos-*` branch matters.

**Agent prompts.** Three static markdown system prompts (`manager/worker-prompt.md`, `orchestrator-prompt.md`, `git-agent-prompt.md`) passed via `--append-system-prompt-file` (`spawner/claude-args.ts:76`). **None mentions worktrees or isolation.** `worker-prompt.md` applies only when `parentId` is set (`manager/routes/workers.ts:62-66`); plain direct-spawned workers get **no system prompt at all**. The `spawn_worker` description says workers run "in your project directory" — false in git repos. Worker reports are free text relayed as `[worker <name> (<id>)] reported:` into the parent's PTY (`manager/routes/workers.ts:348-379`); no handover/test slot exists. No per-spawn dynamic prompt content exists anywhere, but the daemon already synthesizes a per-worker file at spawn (`mcp-<id>.json`, `manager/container.ts:269-287`) and the spawner holds realpath'd `wt.worktreeDir/branch/repoRoot` before the PTY starts.

**Extension points.** Endpoint recipe: Zod schemas + ROUTES in `contracts/src/http.ts` → handler in `manager/routes/` → register in `manager/daemon.ts` → mirror in `manager/web/src/api/routes.js` + `client.js` (deliberate duplication). New event types are enum additions in `contracts/src/events.ts` with no migration; daemon-synthesized events follow the `question_pending` append+publish pattern (`workers.ts:234-239`); log-only events need no `ProcessWorkerEvent` handler. The daemon binds 127.0.0.1:7400 **unauthenticated**, and every spawned agent holds `EOS_DAEMON_URL` — any new endpoint is agent-callable. `core/src/ports/WorktreeManager.ts` + `ChildProcessWorktreeManager` (never-throw `{code,stdout,stderr}` execFile, realpath dance) are the established write-capable git pattern. Events `order=desc` returns newest-N in ASC; the web chat reads only the newest 500 — durable state must not live solely in that window.

## 3. Solution space

Every distinct mechanism proposed, with champion and fate:

1. **Spawn-time env vars** (`EOS_WORKTREE_DIR/BRANCH/SOURCE_ROOT/ISOLATION` in the PTY env, `spawner/worker.ts:90-100`) — all five specialists, identical mechanism. **Adopted unanimously.** Naming settled: `SOURCE_ROOT` (not `REPO_ROOT`, taken by the Eos repo in `container.ts:226`); explicit `ISOLATION=worktree|none` from agent-prompts so non-isolated agents get the negative statement too.

2. **Per-spawn synthesized system-prompt file** — static prompt + generated "Environment" section (literal branch, realpath'd worktree dir, source root, 3-rule isolation contract), gated on **worktreeFrom, not parentId**, so today's prompt-less direct spawns are covered. Championed by four (test-infra initially env-only, conceded in debate). **Adopted.** Location contested: spawner-side `${settings.tmpDir}/system-prompt.md` (agent-prompts) vs daemon-side `container.buildArgs` (worktree-infra, dx, safety originally). Spawner-side won 4–1 in revised positions: facts are realpath'd, the worktree provably exists at write time, tmpDir teardown (`worker.ts:392`) owns cleanup for free, CLI direct spawns covered. Test-infra's dissent (keep prompt assembly at the ADR-0001 choke point) is noted for the backend-parity work.

3. **Prompt/description truth pass** — fix the false `spawn_worker` description; isolation sections in all three prompt files; mandatory "Handover" report line; `git-agent-prompt.md` eos-* topology rules (never `checkout eos-*` — double-checkout refusal; never `branch -D eos-*` — daemon owns that lifecycle). All five. **Adopted.**

4. **Daemon report-relay enrichment** — append `(branch eos-X, worktree <dir>)` to the `[worker … reported:]` header (`manager/routes/workers.ts:362-365`) so the orchestrator holds the merge handle regardless of worker compliance. All five. **Adopted** ("prompts are convention; this line is a guarantee"). Must ship in the same commit as the orchestrator-prompt parse-format update; branch-only fallback during the enrichment window.

5. **Hydrate-at-spawn** (worktree-infra) — APFS clonefile (`cp -c`) of the source checkout's `node_modules` dirs into the worktree, every path gated by `git check-ignore -q` in the source repo (verified: `teardownWorktree`'s `git status --short` dirty check respects .gitignore, so gated hydration cannot false-preserve); symlinks rejected because Node realpath-resolves them back into the user's checkout. **Adopted by majority** as the enabler everyone else's design silently depends on (`/verify` is unfollowable turn-one without deps). Disputed residue: dx wants lazy-on-first-use to avoid taxing every spawn; safety demands `.env*` copying be **opt-in** (secret sprawl into preserved worktrees) — the panel landed on node_modules default-on, `.env*` opt-in, and the synthesized Environment section must reflect the hydration *result* (never claim ".env doesn't exist here" when it was hydrated).

6. **`/verify` action** (agent-prompts, dx, safety) — `verify` in `WorkerActionSchema` + one ACTIONS row + `manager/prompts/verify.md`: run the project's tests/build in YOUR worktree, report with honest verdicts. Executes in the agent's own PTY, so every command rides the existing permission gateway. **Adopted as the v1 verification mechanism.** Test-infra's objection (agent goodwill ≠ evidence) is conceded as true but answered by the verdict chip's honesty model and the deferred deterministic runner.

7. **Verdict vocabulary + chip** (safety) — `passed|failed|blocked|flaky|unverified`, default gray "Unverified," staleness invalidation when edits land after the last verdict. **Adopted unanimously** — best honesty model on the panel ("no news is never green"; "blocked" gives a model an honorable exit, reducing green-washing).

8. **`report_verification` worker-MCP tool** (safety) — structured verdict over MCP. **Deferred**: worker-mcp is wired only when `parentId` is set (`container.ts:265`), so direct-spawned workers — the exact population the prompt-gating fight covered — can't call it. V1 derives the chip by daemon-parsing the Handover/verify transcript output instead.

9. **Daemon merge-back** (worktree-infra; converged shape) — snapshot + `GET /workers/:id/merge/preview` (rev-list ahead/behind + `git merge-tree --write-tree`, ref-only, zero checkout mutation, git<2.38 degrade) + `POST /workers/:id/merge` (clean-user-tree precondition via porcelain, `merge --no-ff`, auto `merge --abort` on conflict, returns `{sha, preMergeHead, lockfileChanged}`), per-repo mutex. **Adopted as the P1 integration primitive** (worktree-infra, agent-prompts, test-infra; safety dropped its ff-only counter).

10. **Temp-index snapshot** (dx-product) — `GIT_INDEX_FILE` + write-tree + commit-tree pinned at `refs/eos/snapshots/<workerId>`: captures uncommitted+untracked work without touching the agent's tree or index, works mid-turn and post-exit. **Adopted unanimously**, replacing worktree-infra's `git add -A && commit` snapshotCommit (**dropped** — mutates a live agent's tree mid-turn, pollutes eos-* history).

11. **ff-only daemon merge** (safety) — **dropped by its own author**: any user commit after spawn (the multi-worker normal case) makes every merge non-ff, routing the common case to a bypassPermissions git agent — a safety inversion. The surviving invariant: the daemon never leaves conflict markers or an unmarked half-state in the user's checkout (satisfied by precondition + merge-tree precheck + auto-abort).

12. **"Try in my checkout"** (dx-product) — `merge --no-commit` ephemeral apply with Keep/Discard, durable `try.json`, shell-level TryBanner; the user's already-running dev server hot-reloads. The strongest product insight (the user's checkout is the only fully-provisioned environment) and the panel's most contested mechanism. **Deferred to P2** under named preconditions: the mid-try-user-edit case is genuinely dangerous (`merge --abort` *refuses* when working-tree edits overlap applied files — stuck MERGE_HEAD, dead Discard button); Keep-then-next-Try is broken as drafted (stashing a half-merge loses merge metadata); one-try-per-repo serializes the multi-agent workflow. Safety conditionally adopted it; the majority deferred it.

13. **Daemon PreviewRunner** (worktree-infra) — managed dev-server runs in the worktree with port lease, `eos-preview-<id>` argv watchdog, stop-before-worktree-remove ordering. The process-lifecycle engineering is the best on the panel and is **kept as the P2 blueprint**, but the trust boundary as proposed is broken: "named targets" resolve from the **worktree's** package.json, which the agent authors — and the mode-resolver 403 checks the *target* worker in the URL, not the caller (an unauthenticated socket has no caller identity). **Deferred** behind the UI token.

14. **Daemon CheckRunner + checks.json** (test-infra) — one-shot deterministic checks, command text only from the user's checkout, runnable post-exit, token-free. The command-text-provenance rule **survives as the precondition for any future runner**, but safety showed the boundary dissolves one dereference down (`npm test` executes the worktree's agent-authored scripts; and `classifyTool` has no path dimension, so an acceptEdits worker can edit `<sourceRepo>/.eos/checks.json` auto-allowed, then curl the endpoint). The honest framing is "executes agent-authored code on the user's explicit click." **Deferred**, human-triggered-only, with an authoring aid to fix cold-start.

15. **Per-boot UI-origin token** (dx-product) — daemon generates a token at startup; the app sends it as a header; required on all new mutating endpoints so agents holding `EOS_DAEMON_URL` cannot self-merge or self-run. **Adopted** as a v1 requirement (worktree-infra and safety converged; the mode-resolver and mtime-heuristic alternatives were both shown unimplementable).

16. **Deletion safety trio** (safety, converged) — kill-confirm dialog with diffstat + unmerged-commit count; `git tag eos/trash/<branch>-<ts36>` before `branch -D` (bytes-cheap in the shared object store, makes deletion recoverable); chat renderer for the existing-but-invisible `worktree {phase:preserved}` event. **Adopted.**

17. **Ambient honesty fixes + escape hatches** (dx-product) — `ComposerDiffRow` chips switch to `gitDirOf` precedence; "isolated · eos-*" chip; Reveal-worktree / Open-in-Terminal / copy-path via existing `/fs/reveal` + `/fs/open`; git-agent escalation preset with the `eos-*` branch passed explicitly (fixing `Composer.jsx:311`). **Adopted** — near-free, correct regardless of centerpiece.

18. **Event hygiene constraints** (test-infra) — 2s/8KB chunk batching, 1MB tail-only caps, never trust the newest-500 window for durable state (type filter on `GET /workers/:id/events` or row persistence). **Adopted as mandatory constraints** for the verdict chip now and any runner later.

## 4. Panel agreements

Unanimous or near-unanimous (all five unless noted):

- PTY env vars carrying realpath'd worktree facts at spawn.
- Per-spawn synthesized system-prompt file gated on `worktreeFrom` (not `parentId`) — 4/5, test-infra conceded in debate.
- Fix the actively false `spawn_worker` description; isolation sections in `orchestrator-prompt.md` and `worker-prompt.md`; mandatory Handover report line.
- Daemon report-relay enrichment as the compliance-independent floor.
- `git merge-tree --write-tree` conflict prediction before any user-checkout mutation; clean-tree precondition; the daemon never writes conflict markers; patch-apply and checkout-of-`eos-*` rejected on git physics (every proposal that touched merge).
- Role:git agent as the **conflict/divergence escalation only**, spawned in the user checkout with the `eos-*` branch passed explicitly — 4/5.
- Kill/delete confirmation + tombstone + rendering the preserved-worktree event — 4/5 designed, fifth endorsed.
- All new events log-only (no `ProcessWorkerEvent` HANDLERS entry), appended via the `question_pending` pattern over existing SSE.
- Routes acting on a worktree must 409 until `worktree_dir` enrichment lands (falling back to `worktree_from` targets the user's repo — the exact bug class being fixed).
- The unauthenticated daemon + agent-held `EOS_DAEMON_URL` makes every new write/exec endpoint an agent-callable consent bypass; a trust answer must ship with the endpoints, not after.
- `DiffViewer` dv-head is the UI anchor; chips must stop describing the wrong repo; escape hatches ship day one.
- Temp-index snapshot as the standard mechanic for capturing uncommitted worktree work.
- Verdict vocabulary with default-Unverified and staleness invalidation.

## 5. Conflicts and resolutions

**C1 — Primary test surface.** Hydrated run-in-place preview (worktree-infra) vs ephemeral apply-to-checkout Try (dx-product, safety-conditional) vs merge-first-then-test (agent-prompts, test-infra) vs checks-only (test-infra original). **Resolution (3–2 in revised positions): merge-back is the v1 accept step**, with `/verify` + hydration covering self-verification and escape hatches covering manual poking. Try is the best-designed fast-follow but its sharpest edge (user edits during an active try → `merge --abort` refusal → stuck MERGE_HEAD) has no specified recovery, and one-try-per-repo serializes Eos's multi-agent premise. PreviewRunner is the right P2 shape but needs the token tier first. **Open:** whether Try or PreviewRunner is the first P2 item — a product call (see §9).

**C2 — Verification executor.** Daemon-run (deterministic, token-free, post-exit-capable) vs agent-PTY-run (inherits the permission gateway). **Resolution: agent-PTY `/verify` in v1.** Safety's caller-identity argument was decisive — both runner trust boundaries (named targets; command-text-from-source-checkout) dissolve one dereference into agent-authored script bodies, and the daemon has no caller identity to authorize against. Deterministic runners return in P2, UI-token-gated and human-triggered, honestly framed. **Open:** whether the verdict chip ever hard-gates merge.

**C3 — Daemon merge semantics.** ff-only (safety) vs `--no-ff` + preconditions + auto-abort (worktree-infra) vs `--no-commit` Try (dx) vs no daemon git writes (agent-prompts original). **Resolution: `--no-ff` with clean-tree precondition, merge-tree precheck, auto-abort, recorded `preMergeHead`.** Safety conflated dirty-tree corruption (excluded by precondition in all designs) with clean-tree conflicted merges (atomically abortable); ff-only would make the Merge button a 409 generator in normal multi-worker use. Safety's crash-window point survives as a documented recovery note (a daemon crash mid-merge can leave MERGE_HEAD; `git merge --abort` recovers; ff's atomicity was the one genuinely load-bearing argument for strictness).

**C4 — Prompt-synthesis location.** Spawner-side (agent-prompts) vs `container.buildArgs` (three originally). **Resolution: spawner-side** (`spawner/prompt-context.ts`) — 4–1 after debate. Test-infra's ADR-0001 choke-point argument is recorded: when in-process backends land, prompt assembly must move behind the backend SPI (InProcessBackend ignores `systemPromptFile` entirely, so non-PTY backends get only env-vars + orchestrator-side awareness until then).

**C5 — Hydration timing and scope.** Every-spawn vs lazy; `.env*` default vs opt-in. **Resolution: node_modules clonefile at spawn** (the agent must be able to self-verify on turn one — that is half of ask (1)); `.env*` strictly opt-in; non-APFS fallback gets a size cap or async path rather than silent minutes before the readiness gate. **Open:** whether monorepo boot latency in practice forces lazy mode.

**C6 — Trust gate.** Per-boot UI token vs target-mode 403 vs mtime heuristics vs pending-approval rows. **Resolution: per-boot UI token, required on `POST /workers/:id/merge` and any future exec endpoint.** The 403 and mtime designs were shown unimplementable on an unauthenticated socket. **Open:** whether merge additionally needs human approval when the panel later allows orchestrator-initiated merges (v1 sidesteps this: UI-only).

**C7 — Verdict channel.** Structured MCP tool vs daemon-parsed Handover line. **Resolution: parsed Handover with fixed vocabulary in v1** (zero contract churn, covers direct spawns); `report_verification` returns in P2 alongside worker-mcp wiring for parentless workers.

## 6. Recommended design

> Decision 2026-06-07: P0 and P1 below are one build, shipped together. P0/P1 labels survive only as logical ordering of the work.

### P0 — Agent awareness (prompt + spawn-time; ship as one unit)

1. **Env vars** — `spawner/worker.ts:90-100` (wt.* already in scope from `setupWorktree`): add `EOS_WORKTREE_DIR`, `EOS_WORKTREE_BRANCH`, `EOS_SOURCE_ROOT` (worktree spawns) and `EOS_ISOLATION=worktree|none` (all spawns). Do not reuse `EOS_REPO_ROOT`.
2. **Synthesized system prompt** — new `spawner/prompt-context.ts`, called between `setupWorktree` and `buildClaudeArgs` in `spawner/worker.ts`: write `${settings.tmpDir}/system-prompt.md` = (static `opts.systemPromptFile` content, if any) + generated `# Environment` section (workerId, name, isolation, realpath'd worktree dir, branch, source root) + 3-rule mini-contract: changes invisible to the user until merged; never instruct the user to run/look at anything in their checkout; self-verify in your own cwd and end reports with a Handover line. `spawner/claude-args.ts:76` already maps the path to `--append-system-prompt-file`; tmpDir teardown (`worker.ts:392`) owns cleanup. Gate on worktree mode. Once P1 hydration ships, condition the "untracked files do not exist here" text on the hydration result.
3. **Prompt truth pass** (one commit):
   - `manager/worker-prompt.md`: "Workspace isolation" section + required 5th report item `Handover: branch <eos-*>; verified by <command + verdict passed|failed|blocked|unverified>; to try after merge: <command>`.
   - `manager/orchestrator-prompt.md`: workers in git repos always run in isolated worktrees; relay the Handover verbatim; never tell the user to run unmerged code; never trust a "verified" claim over a deterministic result; never `kill_worker` before merge or explicit discard.
   - `manager/git-agent-prompt.md`: eos-* topology — merge/cherry-pick from the user checkout; never `git checkout eos-*`; never `branch -D eos-*`; post-merge, tell the user it is now safe to delete the worker.
   - `manager/orchestrator-mcp/tools/spawn_worker.ts` description: replace "in your project directory" with the worktree/eos-branch truth; `manager/worker-mcp/tools/send_message_to_parent.ts` description: add the Handover item.
4. **Relay enrichment** — `manager/routes/workers.ts:362-365`: header becomes `[worker <name> (<id>)] reported (branch <eos-*>, worktree <dir>):` from row fields; branch-only when `worktree_dir` hasn't enriched. Same commit as the orchestrator-prompt parse-format text.

Deploy: `eos restart` (prompt files are read at agent boot; no UI change).

### P1 — Test/verify/merge core

1. **Hydration** — new `spawner/hydrate.ts`, called from `setupWorktree` (`spawner/worktree.ts:52`): find `node_modules` dirs (pruned, depth ≤ 4) in the realpath'd source checkout; each candidate gated by `git check-ignore -q` in the **source** repo (protects the `worktree.ts:77-79` dirty-preservation invariant); `cp -Rc` clonefile, plain-copy fallback with logged warning and size cap; `.env*` only via explicit opt-in config. Hydration summary (per-path ok/skip/fail) rides the `claude_spawning` payload (`spawner/worker.ts:78-84`) and surfaces skips in the UI.
2. **`/verify` action** — `verify` in `WorkerActionSchema` (`contracts/src/http.ts:74`); ACTIONS row in `manager/services/worker-actions.ts:15-20`; new `manager/prompts/verify.md`: detect check commands, run build+tests in YOUR worktree, report per-check pass/fail with exact commands; if required untracked config is missing, verdict `blocked` naming what's missing — never fabricate; mixed re-runs = `flaky`. Buttons in `ComposerDiffRow.jsx` (beside Commit) and `DiffViewer.jsx` dv-head. Rides the existing `/action` → `PromptTemplateService` → displayText pipeline (chat shows `/verify`).
3. **Verdict chip** — fold verify output / Handover lines into `{passed|failed|blocked|flaky|unverified}`; default Unverified; invalidate to Unverified when new edits land after the last verdict (verdict ts vs latest `tool_done`). Derive from events queried directly (add a `type` filter param to `GET /workers/:id/events` rather than trusting the newest-500 chat window). Render in `DiffViewer.jsx` dv-head + `ComposerDiffRow.jsx`.
4. **Try (unstaged apply)** — extend `core/src/ports/WorktreeManager.ts` (or a sibling `BranchIntegration` port; same wiring recipe) with:
   - `snapshot(ref)` — temp-index commit pinned at `refs/eos/snapshots/<workerId>`, agent tree untouched (unchanged from the panel design).
   - `tryPreview(ref)` — rev-list ahead/behind + `git merge-tree --write-tree` conflict prediction; ref-only, zero checkout mutation; degrade below git 2.38.
   - `tryApply(ref)` — preconditions: no other active try in this repo; the **touched files** (paths changed between HEAD and the merged tree) are clean per porcelain — the rest of the tree may be dirty (much weaker precondition than full-clean). Compute patch = `git diff --binary HEAD <mergedTree>`; `git apply` to the working tree only (verify-all-hunks-then-write — near-atomic, index untouched → genuinely unstaged). Persist try state under `~/.eos/tries/<repoKey>/` (workerId, baseHead, patch file, touched files + post-apply blob hashes) so it survives daemon restart and worker deletion. Returns `{files, lockfileChanged}`.
   - `tryDiscard()` — verify each touched file's current hash against the recorded post-apply hash; all intact → `git apply -R <patch>` restores the exact pre-try state (context-anchored, immune to HEAD moving); any file the user edited mid-try → structured 409 listing those files, **nothing reverted** (user commits/stashes/reverts their edits, then retries discard). No MERGE_HEAD, no pending git state — this is what dissolves the panel's stuck-abort objection.
   - `tryKeep()` — delete the try state; the edits remain as the user's own unstaged changes to commit (themselves or via the git agent `/commit`).
   - Per-repo mutex keyed on realpath; one active try per repo (a second Try gets 409 naming the owning worker). Impl in `infra/src/git/ChildProcessWorktreeManager.ts` (existing never-throw + realpath discipline); wire in `manager/container.ts`.

### Implementation notes — deltas from the plan (2026-06-07, as built)

- **No `type` filter on `GET /workers/:id/events`.** The verdict chip derives web-side from the already-loaded event window (`lib/verdict.js`, published via the `pendingQuestion` pattern in selection.jsx). ## 7. User-facing flows

**Flow A — dispatch, verify, try, keep, delete (the happy path).**
The orchestrator dispatches a worker; the spawner creates and hydrates the worktree before claude boots, so the agent can run tests from message one. The agent's system prompt tells it exactly where it is and that the user can't see its files. It works, self-verifies, and reports ending with `Handover: branch eos-fix-auth-…; verified: npm test clean; to try: npm run dev` — and the daemon-stamped header carries the branch even if it forgot. In Eos.app, `ComposerDiffRow` shows the +N/−N badge plus an "isolated · eos-fix-auth" chip that finally describes the same repo as the badge. The user opens the Changes panel, reviews diffs, clicks **Verify** — `/verify` appears in chat, the agent runs the suite in its worktree under the normal permission flow, the chip flips to green "Passed: npm test". The user clicks **Try**: the preview reads "no conflicts, 12 files"; confirm → the daemon snapshots, prechecks the touched files, and applies the merged result as **unstaged edits** in their checkout — their already-running dev server hot-reloads instantly; a TryBanner shows "Trying eos-fix-auth (12 files) — Keep / Discard" plus "lockfile changed — run npm install" when relevant. They poke the app: **Keep** → the edits become their own working-tree changes to commit (or hand to the git agent via `/commit`); **Discard** → exact reverse-apply restores the pre-try state. Deleting the worker afterwards shows the kept/unintegrated status in the confirm, and `eos/trash` tags keep the branch commits recoverable.

**Flow B — conflicts.**
Try preview reports "3 files would conflict." Nothing has touched the user's tree. The panel offers **Resolve with git agent**: a role:git agent spawns in the user's checkout with the prompt naming `eos-fix-auth` explicitly, merges under its AskUserQuestion guard (never checking out or deleting the cm branch), reports success, and says it's now safe to delete the worker.

**Flow C — worker already exited.**
A worker exits naturally with uncommitted changes; the previously invisible "Worktree preserved at `<path>` — N files changed" block now renders in chat with Reveal/Try affordances. Because the row survives and the snapshot/try ops are daemon-side, the user can still review, verify by hand (Open in Terminal into a hydrated worktree), and try — days later, until they delete the row. If they delete without integrating, the confirm warns, and `eos/trash/eos-…` tags keep the commits recoverable.

## 8. Risks and mitigations (ranked)

1. **Agent-callable mutating endpoints (consent/gateway bypass).** The daemon is unauthenticated; agents hold its URL; `classifyTool` has no path dimension, so an acceptEdits worker can auto-allowed-edit any config file and then curl an exec endpoint. *Mitigation:* per-boot UI-origin token on `/try*` and all future checkout-mutating endpoints; no daemon command runner. Target-mode checks and mtime heuristics are rejected as unimplementable.
2. **Silent data loss on delete.** DELETE destroys worktree + branch ~2s post-kill with no confirm today. *Mitigation:* kill-confirm with diffstat + unintegrated count; `eos/trash` tombstone tag before `branch -D`; preserved-worktree event rendered.
3. **Try leaving the user's tree in a bad state.** *Mitigation:* touched-files-clean precondition, merge-tree precheck, `git apply` verifies all hunks before writing (near-atomic, no MERGE_HEAD or pending git state ever), per-repo mutex, structured 409s. Mid-try user edits make Discard refuse per-file with an explicit list instead of corrupting — the user resolves their own edits, then discards. Patch + state persist outside the worktree, so daemon restart or worker deletion never strands an active try.
4. **False-dirty preservation from hydration.** Any non-gitignored hydrated path makes every worktree preserve forever and pressures the prune. *Mitigation:* mandatory `git check-ignore` gate, skip-with-warning surfaced in UI; test against sloppy `.gitignore` repos.
5. **Verdict green-washing.** Verify verdicts are agent-authored; a pressured model may claim `passed`. *Mitigation:* default-Unverified, staleness invalidation, `blocked` as an honorable exit, transcript-visible commands, merge button shows the verdict but doesn't lie about its provenance; deterministic CheckRunner is the P2 systemic fix.
6. **Secret sprawl.** `.env*` hydration copies credentials into worktrees that can outlive the worker. *Mitigation:* opt-in only; preserved-worktree TTL is an open question.
7. **Deps divergence after try/keep.** Agent-installed deps live in the cloned node_modules; the try patch delivers package.json/lockfile only → "works in worktree, broken after keep." *Mitigation:* `lockfileChanged` flag → "run npm install" hint in the TryBanner; not eliminated.
8. **Enrichment window.** Acting before `worktree_dir` lands targets the user's repo — the bug class being fixed. *Mitigation:* 409 + disabled buttons until enrichment.
9. **Event-log durability/bloat.** Verdicts must not depend on the newest-500 window; future runners must not stream megabytes into SQLite. *Mitigation:* type-filter on `/events` for the chip; batching + tail-only caps mandatory for P2 runners.
10. **Custom prompt-file overrides** (`EOS_WORKER_PROMPT_FILE`) silently miss the new static rules. *Mitigation:* the synthesized Environment mini-contract is the per-spawn safety net.
11. **`/verify` permission ping-pong.** Under inherited `default` mode every test command prompts. *Mitigation:* document; surface expected prompts on the Verify button; acceptEdits+ workers are the smooth path.
12. **Non-PTY backends (ADR-0001)** ignore `systemPromptFile`: they get env vars and orchestrator-side awareness only until prompt assembly moves behind the backend SPI.

## 9. Decisions (2026-06-07, maintainer)

1. **Integration gesture** → no v1/later split: build the whole recommended system (P0+P1) as one build. *Initially* `--no-ff` merge-back; **revised same day**: the user wants changes to arrive as **unstaged edits** they can test and then own — so the sole gesture is patch-based Try (apply → Keep/Discard), and the daemon merge commit is dropped. The redesign (merge-tree + `git diff`/`git apply`, no `merge --no-commit`) removes the stuck-MERGE_HEAD edge that originally killed Try in debate.
2. **Trust gate** → per-boot UI token is sufficient. No human-approval pending-row; try stays UI-only (no orchestrator-initiated applies).
3. **Hydration** → as recommended: node_modules clonefile default-on at every worktree spawn; `.env*` strictly opt-in; non-APFS fallback gets size cap + logged warning.
4. **Verification gating** → as recommended: advisory. Try button warns on `failed`/`unverified` but never blocks.
5. **Isolation opt-out** → not needed; `isolation:'none'` is not added. The dead composer worktree chip is **deleted** (P1 item 9).
6. **Deterministic runners** → no. Agent-PTY `/verify` + manual Open-in-Terminal is the long-term stance; PreviewRunner/CheckRunner dropped.
