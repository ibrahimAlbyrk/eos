# Eos iOS Remote Control — Wire & Crypto Protocol Specification

Status: **Implementable v1**. This document turns the settled architecture in
[`ios-remote-control-design.md`](./ios-remote-control-design.md) into byte-exact, cross-platform
implementable bytes. The **architecture is not re-litigated here** — shapes, topologies, capability
tiers, and the threat model come from that document. This spec fixes the *concrete bytes* three
independent implementers (relay, Mac daemon, iOS client) code against so a Swift client and a
Node/TS server interoperate on the first try.

> **Authoring note on consultation.** This spec was to be cross-checked with four peer experts
> (security-auth-architect, connectivity-relay-architect, ios-client-architect,
> eos-internals-expert). The peer-consultation channel was unavailable in this run, so every choice
> below was made directly from (a) the settled design doc, (b) the verified Eos route source
> (`contracts/src/http.ts`, `manager/routes/*`), and (c) primary library/standards research (cited
> inline). Points that genuinely need a second crypto opinion are listed in
> [§12 Unresolved](#12-unresolved-interop-questions), not hidden.

## 0. Conventions

- **Byte order is big-endian** everywhere a multi-byte integer appears on the wire, unless stated.
- **`u8 / u16 / u32 / u64`** = unsigned integers of that width, big-endian.
- **`‖`** = concatenation. **`b64u`** = unpadded base64url (RFC 4648 §5, no `=`). **`b64`** = standard
  base64 with padding. **hex** = lowercase.
- **Label strings** embedded in crypto inputs are **ASCII, no NUL terminator**, exact bytes as quoted.
- All JSON is **UTF-8, no BOM**, object keys as written. JSON is **not canonicalized** except where a
  hash is taken over a body — see [§3.4](#34-body-hashing-for-signatures).
- **"Mac"** = the Eos daemon side (Node). **"device"** = the iOS client (Swift). **"relay"** = the
  dumb forwarder.
- Roles in every handshake: **device = client/initiator**, **Mac = server/responder**. This ordering
  is load-bearing for `crypto_kx` (§1.3) and must never be swapped.

## 1. Crypto suite (LOCKED)

### 1.1 The decision and why it interoperates

The #1 interop risk is two different crypto libraries disagreeing byte-for-byte on the same
primitive. We eliminate it by splitting the suite into two halves, each with a **single, identical
implementation per platform**:

| Layer | Primitive | Node (Mac/relay) | iOS (device) | Why it is byte-identical |
|---|---|---|---|---|
| Session DH, KDF, AEAD | libsodium | `sodium-native` ≥ **5.0.0** | `swift-sodium` (master, SPM) → `Clibsodium.xcframework` | **Same C library** on both ends. crypto_kx / generichash / xchacha20poly1305_ietf are deterministic and version-stable. |
| Static identity signatures | P-256 ECDSA / SHA-256 | Node built-in `crypto` (OpenSSL, `prime256v1`) | CryptoKit `P256.Signing` + `SecureEnclave.P256.Signing` | NIST/SEC1 standard with published KATs; any conformant impl agrees once encoding is pinned (§1.4). |

**Why this split, and not "all libsodium" (Noise/Ed25519) or "all platform-native":**

1. **The device static key MUST live in the Secure Enclave** (design §4.3, §5.4 — non-negotiable).
   The Apple Secure Enclave **only supports NIST P-256** for signing; it **cannot hold an Ed25519 or
   X25519 key** ([Apple: `SecureEnclave.P256.Signing`](https://developer.apple.com/documentation/cryptokit/secureenclave/p256/signing)).
   Therefore the device identity signature is **forced to P-256 ECDSA**. The design doc's literal
   wording "Ed25519 device key in Secure Enclave" is **infeasible as written** and is corrected here.
   This is the one unavoidable deviation from the design's primitive list; everything else honors it.
2. Given the device key is forced to P-256, we make the **Mac static key P-256 too** (symmetry: one
   signature scheme, one verify path on each side, fewer interop surfaces).
3. Everything else — ephemeral DH, key derivation, AEAD — stays in **libsodium**, exactly the
   `X25519 + XChaCha20-Poly1305` choice the design settled. Using the *same* C library on both ends is
   the strongest possible interop guarantee (it is not "two implementations of a spec", it is one
   implementation compiled twice).

This realizes the **same security properties as Noise XX** the design called for — mutual static
authentication + per-session forward secrecy + identity hiding — via an explicit **SIGMA-I**
construction (§2), while avoiding the documented fragility of matching a full Noise-framework
implementation across a Swift lib and a Node lib.

### 1.2 Static identity keys

- **Algorithm:** ECDSA on **NIST P-256 (secp256r1)**, hash **SHA-256**.
- **Mac static identity** (`I_mac`): generated by the daemon, **software-held**. Private key in the
  macOS Keychain (file-mode-0600 PEM acceptable for MVP, per design §4.3). Node:
  `crypto.generateKeyPairSync('ec', {namedCurve:'prime256v1'})`.
- **Device static identity** (`I_dev`): generated **in the Secure Enclave**, non-exportable, biometric
  ACL. Swift: `SecureEnclave.P256.Signing.PrivateKey(accessControl:)` with
  `.privateKeyUsage + .biometryCurrentSet`, persisted as its `dataRepresentation` blob in the Keychain
  (`kSecAttrAccessibleWhenUnlockedThisDeviceOnly`). The private key never leaves the Enclave; signing
  triggers Face ID / Touch ID.
- **Public-key wire encoding (LOCKED):** **SEC1 uncompressed point, 65 bytes** = `0x04 ‖ X(32) ‖ Y(32)`.
  - Swift: `publicKey.x963Representation` (this is exactly `0x04‖X‖Y`).
  - Node: `KeyObject.export({type:'spki'})` → DER; the raw point is the trailing 65 bytes of the
    `subjectPublicKeyInfo` BIT STRING. Implementers MUST convert to/from the 65-byte SEC1 form for the
    wire. Helper: `crypto.createPublicKey({key, format:'der', type:'spki'})` accepts SPKI; to import a
    raw 65-byte point, wrap it as a JWK (`{kty:'EC', crv:'P-256', x:b64u(X), y:b64u(Y)}`) or use the
    SPKI prefix `3059301306072a8648ce3d020106082a8648ce3d030107034200` ‖ point.
- **Signature wire encoding (LOCKED):** **raw `r ‖ s`, 64 bytes** (IEEE-P1363 / fixed-width), **not**
  ASN.1/DER. This is mandatory — DER vs raw is the classic P-256 interop bug.
  - Swift: `signature.rawRepresentation` (64 bytes).
  - Node sign: `crypto.sign('sha256', msg, {key, dsaEncoding:'ieee-p1363'})`.
  - Node verify: `crypto.verify('sha256', msg, {key, dsaEncoding:'ieee-p1363'}, sig)`.
- **What is hashed:** ECDSA-SHA256 hashes the message internally. Callers pass the **raw message
  bytes** (the label-prefixed transcript binding of §2/§3), **not** a pre-hash.

### 1.3 Ephemeral DH + session-key agreement

- **Algorithm:** X25519 via libsodium **`crypto_kx`**. Each side generates a per-session ephemeral
  X25519 keypair with `crypto_kx_keypair` (32-byte public, 32-byte secret). Ephemerals are software
  keys (no Enclave); they are destroyed at session end → **forward secrecy**.
- **Session-key agreement (role-fixed):**
  - Device (client): `crypto_kx_client_session_keys(rx_c, tx_c, ePub_c, eSec_c, ePub_s)`.
  - Mac (server): `crypto_kx_server_session_keys(rx_s, tx_s, ePub_s, eSec_s, ePub_c)`.
  - libsodium guarantees `tx_c == rx_s` and `rx_c == tx_s`. Define the two 32-byte shared keys:
    - **`K_c2s` (device→Mac)** `= tx_c = rx_s`
    - **`K_s2c` (Mac→device)** `= rx_c = tx_s`
- Internally `crypto_kx` computes `BLAKE2b-512( X25519(eSec, otherEPub) ‖ ePub_client ‖ ePub_server )`
  and splits into `rx‖tx`. This is deterministic and identical across all libsodium builds, which is
  why it is safe to rely on as the cross-platform KDF for the DH step.
- swift-sodium high-level form:
  `sodium.keyExchange.sessionKeyPair(publicKey: ePub_c, secretKey: eSec_c, otherPublicKey: ePub_s, side: .CLIENT)`
  → `.rx` / `.tx`.

### 1.4 KDF (transcript-bound key derivation)

A single KDF function, used for every derived key:

```
KDF(key, label, transcriptHash) = crypto_generichash(
    outlen = 32,
    in     = label ‖ transcriptHash,     // label = ASCII bytes; transcriptHash = 32 bytes
    key    = key                          // 32-byte BLAKE2b key
)
```

- `crypto_generichash` is **BLAKE2b**; keyed mode with a 32-byte key, 32-byte output.
- Node: `sodium.crypto_generichash(out32, input, key32)`.
- Swift: `sodium.genericHash.hash(message: input, key: key32, outputLength: 32)` — or call
  `crypto_generichash` from `Clibsodium` directly for an exact match (the high-level wrapper is a thin
  shim over the same C call).
- **Transcript hash** `TH = crypto_generichash(32, in = <concatenated handshake bytes>, key = none)`
  (unkeyed BLAKE2b-256). The exact concatenation per handshake is in §2.

### 1.5 AEAD (per-frame record protection)

- **Algorithm:** **XChaCha20-Poly1305-IETF** — `crypto_aead_xchacha20poly1305_ietf_encrypt/decrypt`.
- **Key:** 32 bytes (a derived traffic key, §2.5).
- **Nonce (LOCKED, 24 bytes):**

  ```
  nonce[0]      = epoch      (u8)   key-epoch, 0x00 at session start
  nonce[1]      = dir        (u8)   0x00 = device→Mac (c2s), 0x01 = Mac→device (s2c)
  nonce[2..10)  = seq        (u64 BE)  per-(direction,epoch) monotonic counter, starts at 0
  nonce[10..24) = 0x00 × 14         reserved, must be zero
  ```

  XChaCha's 24-byte nonce comfortably holds this structured counter. Because `seq` is a strictly
  increasing per-direction counter that **never repeats under a given (key, epoch)**, the nonce is
  collision-free by construction (no randomness needed). A `seq` rollover (2⁶⁴) or any reuse MUST force
  a rekey/epoch bump before encrypting (practically unreachable).
- **AAD (LOCKED):** the AEAD authenticates the relay-visible routing header so a malicious relay
  cannot tamper with routing without detection:

  ```
  AAD = ver(1) ‖ epoch(1) ‖ dir(1) ‖ seq(8) ‖ roomLen(1) ‖ room(roomLen) ‖ clientId(16)
  ```

  These are exactly the outer-envelope fields (§3.1). `seq`/`dir`/`epoch` in the AAD MUST equal those
  in the nonce; a mismatch is a `DECRYPT_FAIL`.
- **Ciphertext = `tag(16) ‖ encrypted-bytes`?** No — libsodium's combined mode appends the tag:
  output is **`encrypted-bytes ‖ tag(16)`** (length = plaintext_len + 16). This is the on-wire payload.
- **Caller-supplied nonce note (iOS):** swift-sodium's high-level
  `aead.xchacha20poly1305ietf.encrypt(...)` **auto-generates and prepends** a random nonce — **do NOT
  use it.** Our nonce is deterministic and is carried in the header, not prepended. Call the C function
  from `Clibsodium` directly:
  `crypto_aead_xchacha20poly1305_ietf_encrypt(c, &clen, m, mlen, ad, adlen, nil, npub, k)` with our
  24-byte `npub`. Node's `sodium-native` already takes an explicit `npub` buffer.
- **Forward-secrecy posture:** per-session ephemeral X25519 gives FS across sessions. **Within** a
  session there is no automatic rekey in v1 (the `epoch` byte is reserved for a Phase-2 in-session
  rekey). A session compromise exposes that session's traffic only after the compromise; prior sessions
  remain protected because their ephemerals are gone.

## 2. Handshake state machines

Three handshakes share the same primitives. All handshake frames are **inner frames** (§4) of type
`hs` / `resume`, carried inside the transport (LAN WS or relay envelope). The relay forwards them
opaquely; on LAN they ride the WS directly.

### 2.1 Cold PAIRING (first contact, one-time-secret + enrollment)

Pre-state: the device has scanned the **QR** (§6) and therefore holds, pre-authenticated:
`I_mac_pub` (pinned), `ots` (32-byte one-time secret), the LAN/relay addresses, the LAN cert SPKI pin,
and a one-time **pairing bearer** to reach the room.

Messages (device = client, Mac = server):

```
PAIR-1  device → Mac     { v:1, t:"hs", step:1, mode:"pair",
                           ePubC: b64u(ePub_c[32]),
                           nC:    b64u(clientNonce[16]) }

PAIR-2  Mac → device      { v:1, t:"hs", step:2, mode:"pair",
                           ePubS: b64u(ePub_s[32]),
                           nS:    b64u(serverNonce[16]),
                           encS:  b64u( AEAD(K_hs_s2c, N0_s2c, AAD=∅, P=S2) ) }
                          where S2 = { iMac: b64u(I_mac_pub[65]), sigS: b64u(sig_s[64]) }

PAIR-3  device → Mac      { v:1, t:"hs", step:3, mode:"pair",
                           encC:  b64u( AEAD(K_hs_c2s, N0_c2s, AAD=∅, P=C3) ) }
                          where C3 = { iDev: b64u(I_dev_pub[65]),
                                       devId: "<uuid>", label: "<string>",
                                       sigC:  b64u(sig_c[64]),
                                       ots:   b64u(otsProof[32]) }
```

Derivations and checks, in order:

1. After PAIR-1/PAIR-2 both sides run `crypto_kx` (§1.3) → `K_c2s`, `K_s2c`.
2. `TH2 = crypto_generichash(32, in = ePub_c ‖ clientNonce ‖ ePub_s ‖ serverNonce)`.
   Handshake key `K_hs_s2c = KDF(K_s2c, "eos/v1 hs s2c", TH2)`.
3. **Mac signs** `sig_s = ECDSA-P256-SHA256( I_mac_priv, "eos/v1 pair server" ‖ TH2 )` and seals `S2`
   under `K_hs_s2c` with nonce `N0_s2c` (= nonce with epoch 0, dir s2c, **seq 0**).
4. **Device** opens `encS`, asserts `iMac == pinned I_mac_pub` (**fail → abort: MITM**), verifies
   `sig_s` over `"eos/v1 pair server" ‖ TH2`.
5. `TH3 = crypto_generichash(32, in = ePub_c ‖ clientNonce ‖ ePub_s ‖ serverNonce ‖ encS)`.
   `K_hs_c2s = KDF(K_c2s, "eos/v1 hs c2s", TH3)`.
6. **Device computes the one-time-secret proof** `otsProof = crypto_generichash(32, in = TH3, key = ots)`
   (keyed BLAKE2b — proves a human scanned *this* QR and bound it to *this* handshake).
7. **Device signs** (Face ID) `sig_c = SE-ECDSA-P256-SHA256( I_dev, "eos/v1 pair client" ‖ TH3 )` and
   seals `C3` under `K_hs_c2s` (nonce seq 0, dir c2s).
8. **Mac** opens `encC`, verifies `otsProof` (recompute, constant-time compare; **single-use** — burn
   `ots` immediately), verifies `sig_c` over `"eos/v1 pair client" ‖ TH3` against `iDev`. On success:
   - **Enroll** `iDev` + `devId` + `label` into `~/.eos/devices` (file mode 0600).
   - Issue the **durable per-device bearer** and the **first resumption ticket** (§2.4) in a
     post-handshake `control`-reply / server frame, sealed with the now-active traffic keys (§2.5).
   - Update the relay allowlist: add `SHA-256(durable bearer)`, drop the pairing bearer hash.

Pairing endpoint MUST be heavily rate-limited / lockout-protected (design §4.6).

### 2.2 Cold CONNECT (post-pairing, issues a resumption ticket)

Identical message flow to PAIRING with `mode:"connect"` and **two differences**:

- **No `ots`** field; the Mac authorizes by looking `iDev`/`devId` up in the allowlist (`~/.eos/devices`)
  rather than enrolling. An unknown `iDev` → `AUTH_FAILED`, force re-pair.
- Labels in the signed strings change to `"eos/v1 conn server"` / `"eos/v1 conn client"`.
- Face ID is **required** (fresh SE signature) — this is the "cold handshake = Face ID" state.

On success the Mac issues a **fresh resumption ticket** (§2.4) over the established channel.

### 2.3 RESUME (warm reconnect, PSK-(EC)DHE, no biometric)

Pre-state: the device holds a valid, unexpired resumption ticket `{ticketId, PSK}` (Keychain
`WhenUnlockedThisDeviceOnly`, no biometric ACL). Two messages, **no P-256 signature, no Face ID**:

```
RES-1  device → Mac    { v:1, t:"resume",
                         ticketId: b64u(ticketId[16]),
                         ePubC:    b64u(ePub_c[32]),
                         nC:       b64u(clientNonce[16]),
                         binder:   b64u(bC[32]) }
       bC = crypto_generichash(32, in = "eos/v1 resume binderC" ‖ ticketId ‖ ePub_c ‖ clientNonce,
                               key = PSK)

RES-2  Mac → device    { v:1, t:"resume-ok",
                         ePubS:    b64u(ePub_s[32]),
                         nS:       b64u(serverNonce[16]),
                         binder:   b64u(bS[32]),
                         encTicket: b64u( AEAD(K_resume_ticket, nonce(epoch0,s2c,seq0), AAD=∅, P=newTicket) ) }
       bS = crypto_generichash(32, in = "eos/v1 resume binderS" ‖ ticketId ‖ ePub_s ‖ serverNonce ‖ ePub_c,
                               key = PSK)
```

Flow:

1. Mac looks `ticketId` up in authoritative ticket state. Unknown/expired → `TICKET_INVALID`.
   **Already-consumed → `TICKET_REUSE`**: invalidate the whole ticket *family*, force a cold handshake,
   raise an audit alert (design §3.4 single-use rotation).
2. Both verify the peer binder (constant-time). A bad binder → `AUTH_FAILED` (proves PSK possession =
   mutual auth without signatures).
3. `crypto_kx` over the fresh ephemerals → `K_c2s`, `K_s2c`.
   `TH = crypto_generichash(32, in = ticketId ‖ ePub_c ‖ clientNonce ‖ ePub_s ‖ serverNonce)`.
4. **PSK-(EC)DHE mixing** — the traffic keys mix BOTH the fresh DH and the ticket PSK, so resume is
   forward-secret *and* ticket-bound:

   ```
   K_c2s_final = KDF( key = PSK, label = "eos/v1 resume data c2s", TH_with_kx )
   K_s2c_final = KDF( key = PSK, label = "eos/v1 resume data s2c", TH_with_kx )
   K_resume_ticket = KDF( key = PSK, label = "eos/v1 resume ticket", TH_with_kx )   // seals encTicket only
   where TH_with_kx = crypto_generichash(32, in = TH ‖ K_c2s ‖ K_s2c)
   ```

   **encTicket seal (LOCKED).** `encTicket` is sealed with the **dedicated** key `K_resume_ticket`
   (NOT `K_s2c_final`), nonce `epoch=0 ‖ dir=0x01 (s2c) ‖ seq=0`, and **`AAD = ∅`** (empty). The
   dedicated key is deliberate: it keeps the encTicket out of the `K_s2c_final` (key, nonce) space so
   it cannot collide with the s2c data stream's own `seq 0` frame — i.e. **both** `K_c2s_final` and
   `K_s2c_final` data streams start at `seq 0` after resume, with no nonce reuse. AAD is empty (matching
   the cold-handshake `encS`/`encC` seals, §2.1): `K_resume_ticket` is already uniquely bound to this
   handshake (PSK + fresh ephemerals + `TH_with_kx`), so an empty AAD is safe and a captured `encTicket`
   is undecryptable in any other session.

5. Mac consumes the ticket, issues a **new** `{ticketId', PSK'}` of the same family inside `encTicket`,
   device replaces its stored ticket. Capabilities of a resumed session are **read + low-risk only**
   (§5); high-risk verbs still require per-action SE step-up (§7.3).

### 2.4 Resumption ticket (contents + binding)

The ticket is **Mac-authoritative**. The device stores only `{ticketId, PSK}`.

```
Ticket (Mac-side record, never sent whole to device):
  ticketId   : 16 bytes random            (the lookup handle)
  familyId   : 16 bytes random            (rotation lineage; reuse invalidates the whole family)
  PSK        : 32 bytes random            (resumption pre-shared key)
  devId      : string                     (bound to one enrolled device/static key)
  caps       : ["read","lowrisk"]         (NEVER includes high-risk verbs)
  epoch      : u8
  issuedAt   : unix seconds
  idleExp    : issuedAt + 24h  (sliding; refreshed each successful resume)
  absExp     : familyOrigin + 7d (hard ceiling, not refreshed)
  consumed   : bool                       (single-use; set on first RES-1 accept)
```

Sent to the device (inside `encTicket`, or post-cold-handshake): `{ ticketId, PSK, idleExp, absExp }`.

**Bindings:** a ticket is cryptographically useless without (a) the PSK (held only by the legit device
+ Mac) and (b) being addressed to the device's enrolled `devId`. It is **capability-bounded to
read+low-risk**, **time-bounded** (sliding 24h idle + absolute 7d), and **single-use** (consumed on
redemption, family-invalidated on reuse). The Mac exposes a **panic "invalidate all tickets"**
(design §3.4). A stolen ticket therefore reaches **read-only state and never RCE**.

### 2.5 Traffic keys (post-cold-handshake)

After a successful PAIRING or CONNECT (`TH3` final):

```
K_c2s_final = KDF( key = K_c2s, label = "eos/v1 data c2s", TH3 )
K_s2c_final = KDF( key = K_s2c, label = "eos/v1 data s2c", TH3 )
```

Data frames (`event`/`patch`/`snapshot`/`reply`/`control`/`ka`) use these keys with the §1.5 nonce,
**`seq` starting at 0** per direction under `epoch 0`. Handshake identity frames used distinct
`K_hs_*` keys, so there is no key/nonce overlap. The two directions have independent `seq` counters.

## 3. Signing & hashing details

### 3.1 What each handshake signature covers

| Signer | Message signed (raw bytes, ECDSA-P256 internally SHA-256s it) | Proves |
|---|---|---|
| Mac (PAIR-2/CONNECT-2) | `"eos/v1 pair server"\|"eos/v1 conn server" ‖ TH2` | This is the real Mac (key pinned via QR), bound to these ephemerals |
| Device (PAIR-3/CONNECT-3) | `"eos/v1 pair client"\|"eos/v1 conn client" ‖ TH3` | Possession of the Secure-Enclave device key, bound to the full transcript |

`TH2`/`TH3` are the BLAKE2b-256 transcript hashes defined in §2.1. Signing the transcript (not just a
nonce) binds identity to the exact ephemeral keys → SIGMA-I mutual authentication; sealing the
identity+signature under the derived `K_hs_*` key gives identity hiding from the relay/observer.

### 3.2 Step-up signature (high-risk control)

See §7.3 for the full sub-protocol. The signed message is:

```
stepUpMsg = "eos/v1 stepup" ‖ "\n" ‖ hex(sessionTH) ‖ "\n" ‖ method ‖ "\n" ‖ path ‖ "\n"
            ‖ hex(bodyHash) ‖ "\n" ‖ b64u(challengeNonce) ‖ "\n" ‖ decimal(ts)
sig_su = SE-ECDSA-P256-SHA256( I_dev, stepUpMsg )
```

- **`sessionTH` (session binding, LOCKED):** the 32-byte handshake transcript hash that derived **this
  live E2E session's** traffic keys — `TH3` for a cold PAIR/CONNECT session (§2.1/§2.2), or `TH_with_kx`
  for a RESUME session (§2.3). Both the device and the daemon already hold this value from the handshake
  that established the session; neither transmits it. Folding it into the signed message binds the
  step-up to the exact handshake transcript (and thus the exact ephemeral keys / session), so a captured
  `sig_su` cannot be replayed into a *different* session even by the same enrolled device. This is
  defense-in-depth layered over the single-use `challengeNonce`: the nonce stops in-session replay, the
  `sessionTH` binding stops cross-session transplant. The daemon verifies `sessionTH` equals the
  transcript hash of the session the frame arrived on (§7.3 step 4); a resumed (ticket) session cannot
  satisfy step-up at all (no Enclave key), so `sessionTH` there only ever applies to a cold session that
  later performs a high-risk action.

### 3.3 Resume binders

Keyed BLAKE2b (`crypto_generichash` with `key=PSK`), defined inline in §2.3. Not ECDSA.

### 3.4 Body hashing for signatures (LOCKED — `body` is an opaque string)

The `control` frame's **`body` is transmitted as a JSON *string*** (not a nested JSON object): its
UTF-8 bytes ARE the request-body serialization the route handler will parse. Carrying the body as an
opaque string is what makes "the exact transmitted bytes" unambiguous — there is no nested sub-object
whose byte boundaries are lost when the outer inner-frame is parsed, so **no canonicalization is ever
required.**

```
bodyHash = crypto_generichash(32, in = <the UTF-8 bytes of the DECODED `body` string value>)
```

- **Device (signer).** Builds the body serialization `bodyStr` (any valid JSON for the route — the
  device chooses the exact bytes), computes `bodyHash` over `UTF-8(bodyStr)`, sets `body: bodyStr`
  (a JSON string) in the control frame, and signs `stepUpMsg` carrying `hex(bodyHash)` (§3.2).
- **Daemon (verifier).** After AEAD-decrypting and parsing the inner frame, it reads `body` as a
  **string value**, hashes that string's UTF-8 bytes **verbatim** (NO `JSON.stringify`, no
  re-serialize), compares to the signed `bodyHash`, and **only then** `JSON.parse`s `bodyStr` into the
  object dispatched into the route handler. The daemon MUST NOT hash a re-serialized parse of the body.
- **Why this is byte-exact across platforms.** JSON string encode/decode preserves the string's
  code-point content exactly (escaping is non-unique but *decoding* is), so the device's pre-escape
  bytes and the daemon's post-decode bytes are byte-identical regardless of on-wire escaping. This
  removes the key-order / number-format / whitespace mismatch that hashing a re-serialized *object*
  would introduce.
- READ/LOW control frames also carry `body` as a string (uniform frame shape) but compute no
  `bodyHash`. For a GET with no body, `body = "{}"`.

## 4. Frame formats

Two layers: the **outer envelope** (relay-visible, binary) and the **inner frame** (E2E-encrypted
JSON). On **LAN-direct** there is no relay; the outer envelope is still used (so the codec is identical
in both modes), but the daemon both terminates TLS and is the routing target.

### 4.1 Outer envelope (relay-visible, binary, big-endian)

One envelope = one WebSocket **binary** message.

```
Offset  Size  Field        Notes
0       1     ver          0x01
1       1     type         0x01 data | 0x02 register | 0x03 join | 0x04 relayctl | 0x05 ka | 0x06 error
2       1     dir          0x00 c2s | 0x01 s2c
3       1     epoch        key-epoch (0x00 in v1)
4       8     seq          u64 BE  (per direction+epoch)
12      1     roomLen      u8  (room is 22 ASCII chars in v1 → fits)
13      R     room         ASCII (R = roomLen)
13+R    16    clientId     16 bytes; all-zero for register and for Mac→relay broadcasts
29+R    ..    payload      remainder of the WS message
```

- For `type=0x01 data`: `payload` is the **opaque AEAD ciphertext** (`encrypted ‖ tag16`). The relay
  forwards it verbatim; it never parses or decrypts it. The header fields `{ver,epoch,dir,seq,room,
  clientId}` are exactly what the AEAD binds as AAD (§1.5), so the relay cannot tamper with routing.
- For `type` ∈ {`register`,`join`,`relayctl`,`ka`,`error`}: `payload` is a **UTF-8 JSON object the
  relay DOES parse** (these are relay-control, not E2E). See §5.
- **`sizes`** in the design's `{room,clientId,dir,epoch,seq,sizes}` = the WS message length, which the
  relay sees inherently; no separate field is needed.
- **Max envelope size:** 5 MiB (relay rejects larger with `error` ⇒ `FRAME_TOO_LARGE`).

### 4.2 Inner frame (E2E plaintext = the AEAD plaintext)

The decrypted AEAD plaintext is **exactly one UTF-8 JSON object** = one inner frame. No length prefix
inside the AEAD (the ciphertext length defines the boundary). The **type tag is the `t` field.**

server → client:

```
event    { t:"event",    seq, reason, ts, payload }      // mirrors SSE change + seq cursor; reason = bus topic verbatim
patch     { t:"patch",    seq, resource, op:"upsert"|"remove", data }
snapshot  { t:"snapshot", seq, workers, pending }         // cold start / after a gap
reply     { t:"reply",    correlationId, status, body }   // response to a control call (status = HTTP-ish int)
ka        { t:"ka", ts }                                  // keepalive
challenge { t:"challenge", challengeNonce, expiresAt }    // step-up challenge issuance (§7.3)
error     { t:"error", code, message, correlationId? }    // §7.2
```

client → server:

```
hello    { t:"hello",   lastContentId, resumptionTicket?, resumeEphemeralPub? }   // opening frame
control   { t:"control", correlationId, method, path, body, stepUp? }              // body = JSON STRING (§3.4)
ka        { t:"ka", ts }
```

- `hs` / `resume` / `resume-ok` frames (§2) are also inner frames but are exchanged **before** traffic
  keys exist; they are sealed with `K_hs_*` (cold) or are partly cleartext + binder (resume) as defined
  in §2, then carried in the same outer envelope (`type=0x01`, but the relay still never inspects them).
- **`control`** is a tunneled REST call dispatched into the existing `manager/routes/*` handlers via
  the design's virtual-response shim. `method` ∈ `GET|POST|PUT|DELETE`; `path` is one of the ROUTES
  strings (§5); `body` is a **JSON string** holding the route's request body serialized (`"{}"` for
  GET). It is carried opaquely so its exact bytes can be hashed for step-up (§3.4); the daemon hashes
  it first, then `JSON.parse`s it for dispatch. (`reply.body`, server→client, is a normal JSON value —
  it is never signed.) When `path` needs query parameters, classify the **path portion only** (split on
  `?`) against the §8 tiers, then dispatch the full `path` incl. query — a query string MUST NOT defeat
  tier classification.
- **`stepUp`** present only on high-risk control frames (§7.3).
- **Max inner plaintext:** 1 MiB. A `snapshot` that would exceed this MUST be split: send a lean
  `snapshot` (ids + minimal fields) then page detail via `control` GETs / `patch` frames (mirrors the
  web `eventsStore` `order:desc limit 500` + `beforeId` paging). Implementations MUST NOT emit a single
  >1 MiB frame.

### 4.3 Correlation & dedup

- `correlationId`: client-generated, **UUIDv4 string**, unique per in-flight `control`; the matching
  `reply` echoes it. Client keeps a `correlationId → continuation` map with a timeout (design §5.2).
- `clientMsgId` (inside a message body): dedups message retries across reconnects.

## 5. Relay protocol (pure routing, never decrypts)

The relay is a dumb unicast forwarder. It parses only `register`/`join`/`relayctl`/`ka`/`error`
payloads and the **outer header** of `data` frames; it never decrypts E2E payloads.

### 5.1 Identifiers & credentials

- **Room id:** 16 random bytes → **`b64u` (22 chars)**. Opaque, generated by the Mac at remote-enable,
  carried in the QR. Used as the routing key.
- **Per-device bearer:** **256-bit (32 bytes) random → `b64u` (43 chars)**. Carried in the device's WS
  upgrade header `Authorization: Bearer <b64u>` AND in the `join` payload (for relay admission). The
  **relay stores ONLY `SHA-256(bearer)`** (32-byte hash) in the room allowlist; admission is a
  **constant-time compare** of `SHA-256(presented bearer)` against the stored hashes. The relay never
  holds a usable bearer at rest.
- **Mac room-owner secret:** 256-bit random → `b64u`; the relay stores only its `SHA-256`. Presented in
  `register` to claim/own a room.

### 5.2 register (Mac → relay), JSON payload of a `type=0x02` envelope

```json
{ "t":"register", "room":"<b64u22>", "owner":"<b64u bearer>",
  "allow":["<sha256 hex of device-bearer>", ...] }
```

- Relay verifies `SHA-256(owner)` against the room's stored owner hash (first registration *sets* it,
  persisted by the operator out-of-band or pinned on first use — see §12). Binds this socket as the
  room's **Mac endpoint**. Re-register replaces the socket (Mac reconnect).
- `allow` is the current device-bearer-hash allowlist; the Mac may update it later via `relayctl`
  (`{t:"allow-add"|"allow-remove", room, hash}`) — e.g. add a durable bearer + drop the pairing bearer
  after enrollment.

### 5.3 join (device → relay), JSON payload of a `type=0x03` envelope

```json
{ "t":"join", "room":"<b64u22>", "bearer":"<b64u43>", "apnsToken":"<hex>"? }
```

- Relay computes `SHA-256(bearer)`, constant-time-checks membership in the room's `allow` set. Miss →
  `error` ⇒ `BEARER_DENIED`, close. Hit → assign a **`clientId` (16 random bytes)**, register the
  device socket under `room → devices[clientId]`, and notify **both** ends of the assignment:
  - **→ Mac**, via `relayctl` `{t:"joined", clientId, room}`, so it can start its per-device E2E
    handshake (and key the session by `clientId`).
  - **→ the joining device**, via a `relayctl` **join-ack** (`type=0x04`, `dir=0x01 s2c`,
    `clientId` = the just-assigned value) carrying the same `clientId` (see join-ack frame below).
    The device MUST receive this before it can emit any frame: `clientId` is a mandatory outer-header
    field on every device→Mac envelope (§4.1) **and** an authenticated AAD input (§1.5), so without
    it the device can neither address its `data` frames nor compute/verify AEAD AAD. The device MUST
    NOT send `hs`/`resume`/`data` frames until the join-ack arrives.
- `apnsToken` (optional) is stored for opt-in push egress (design §5.5/§7.4); absent → no push.

**join-ack (relay → device), JSON payload of a `type=0x04 relayctl` envelope:**

```json
{ "t":"joined", "room":"<b64u22>", "clientId":"<b64u of 16-byte clientId>" }
```

The relay generates `clientId` and is the sole authority for it; the device treats the received value
as opaque and copies it verbatim into the outer header (§4.1) of every frame it sends. The same
`clientId` the Mac learns from its `{t:"joined"}` notification and the device learns from this join-ack
MUST be byte-identical — it is the per-device routing + AAD key for the whole session. (On **LAN-direct**
there is no relay; the daemon assigns and returns the `clientId` in the same join-ack shape over the WS
before any handshake frame.)

**`clientId` encoding (LOCKED).** In **both** `{t:"joined"}` JSON bodies (relay→Mac notify and
relay→device join-ack) the `clientId` is **`b64u`** — unpadded base64url of the 16 raw bytes (22 chars),
never hex. On the relayctl envelopes themselves (`type=0x04`), the relay **stamps the assigned 16-byte
`clientId` into the outer-header `clientId` field** (offset 13+R): the all-zero header rule of §4.1
applies **only** to `register` and Mac→relay broadcasts, not to per-device relayctl. `relayctl` frames
are not AEAD-protected, so the outer-header `clientId` on them is a routing / cross-check aid (the device
SHOULD assert header == JSON), **not** an AAD input — the load-bearing copy is the b64u value in the JSON
body, which both ends decode to the same 16 raw bytes.

### 5.4 Routing rules

- `data` frame from a device (`dir=c2s`) → forward verbatim to the room's **Mac** socket, preserving
  the envelope (the Mac reads `clientId` to pick the right per-device key).
- `data` frame from the Mac (`dir=s2c`) → forward verbatim to `devices[clientId]`. **Pure unicast** —
  because E2E is pairwise, the Mac emits **N per-device ciphertexts**, each addressed by `clientId`.
- `ka` frames are forwarded (or answered locally — relay choice; both ends tolerate either).
- The relay holds an **in-memory room registry only**; it persists no plaintext, no keys, no content.
- The relay is **Eos-protocol-version-independent**: it never parses inner frames, so protocol
  evolution requires no relay upgrade.

### 5.5 Relay errors (`type=0x06`, cleartext JSON)

`{ "t":"error", "code":"ROOM_NOT_FOUND"|"BEARER_DENIED"|"OWNER_MISMATCH"|"ROOM_FULL"|"FRAME_TOO_LARGE"|"RATE_LIMITED", "message":"..." }`

## 6. Pairing QR payload

A single JSON object, shown as a QR by the Mac app. Compact keys to keep the QR dense.

```json
{
  "v": 1,
  "typ": "eos-pair",
  "macPub":  "<b64u of I_mac_pub, 65-byte SEC1>",
  "ots":     "<b64u of 32-byte one-time pairing secret>",
  "otsExp":  1750600000,
  "lan":     ["wss://192.168.1.42:7400/ws", "wss://[fe80::...]:7400/ws"],
  "lanSpki": "<b64 of SHA-256(DER SubjectPublicKeyInfo of the LAN self-signed cert)>",
  "relay":   { "url": "wss://ibrahim-albyrk.zap.cloud:443", "room": "<b64u22>" },
  "bearer":  "<b64u of 32-byte ONE-TIME PAIRING bearer>",
  "exp":     1750600000
}
```

- `macPub` is the **pinned** Mac identity (the device asserts equality in PAIR-2).
- `ots` is single-use, short-lived (`otsExp`, e.g. now+120s); consumed at PAIR-3.
- `bearer` admits the device to the relay room **for the pairing handshake only**. After enrollment the
  Mac issues the durable per-device bearer over E2E and swaps the relay allowlist (§5.2). On LAN-only
  pairing `relay`/`bearer` may be `null`.
- **Cert pinning scope (per the confirmed real domain):** the **relay leg** uses the public domain
  `ibrahim-albyrk.zap.cloud` with standard **ACME/public-CA TLS** → normal certificate validation,
  **no pinning**. **Pinning applies ONLY to the LAN self-signed leg** via `lanSpki` (the device's
  `URLSessionDelegate` compares the server cert's SPKI-SHA-256 to `lanSpki`). This matches design §5.4.
- The QR is consumed once; a second scan with a burned `ots` fails closed.

## 7. Versioning, errors, and the step-up sub-protocol

### 7.1 Versioning

- **Protocol version** is `1` in: the outer `ver` byte (`0x01`), the QR `v`, and every handshake frame
  `v`. The device announces its version in PAIR-1/CONNECT-1/RES-1; the Mac rejects an unsupported
  version with `error ⇒ BAD_VERSION` and closes. Minor additive fields MUST be ignored if unknown
  (forward-compatible); a breaking change bumps `ver` to `0x02`.

### 7.2 Error frames (inner, E2E)

`{ t:"error", code, message, correlationId? }`. Codes (LOCKED enum):

```
BAD_VERSION       unsupported protocol version
AUTH_FAILED       signature / allowlist / binder check failed
DECRYPT_FAIL      AEAD verify failed or AAD/nonce mismatch
REPLAY            seq not strictly increasing / nonce reuse detected
SEQ_GAP           receiver detected a gap → peer should request a snapshot
TICKET_INVALID    unknown/expired resumption ticket
TICKET_REUSE      consumed ticket re-presented → family invalidated, cold handshake forced
STEPUP_REQUIRED   high-risk control without a valid step-up signature
STEPUP_INVALID    step-up signature/nonce/binding check failed
CAP_DENIED        device capability tier does not permit this route
ROUTE_REFUSED     route is never exposed remotely (§8)
RATE_LIMITED      per-device or global rate limit hit
INTERNAL          unexpected server error
```

### 7.3 Step-up sub-protocol (high-risk control)

Every **HIGH-RISK** control (§8) requires a **fresh per-action Secure-Enclave signature** proving a
live biometric authorization of *this exact action*. Flow:

1. **Challenge.** The device requests a challenge: `control { method:"POST", path:"/stepup/challenge",
   body:{} }` (a new daemon-side virtual route, not a worker route). The daemon returns a `challenge`
   frame `{ challengeNonce: b64u(16 random bytes), expiresAt }` and records it as **issued, unconsumed,
   single-use, short TTL (≤60s)**. (Alternatively the daemon may pre-issue a rolling `challenge` with
   each `reply`; the on-demand form is the normative one.)
2. **Sign.** The device builds `stepUpMsg` (§3.2) over the **actual** `method`/`path`/`bodyHash`/
   `challengeNonce`/`ts`, signs with the Enclave key (**Face ID/Touch ID prompt**) → `sig_su` (64-byte
   raw r‖s).
3. **Send.** The high-risk `control` frame carries:

   ```json
   "stepUp": { "challengeNonce":"<b64u16>", "ts": 1750600000, "sig":"<b64u of sig_su[64]>" }
   ```

4. **Verify (daemon).** Accept iff ALL hold:
   - `challengeNonce` is known, unconsumed, unexpired → **consume it** (single-use, prevents replay);
   - `sessionTH` reconstructed in `stepUpMsg` (the daemon uses **its own** stored transcript hash for the
     session this frame arrived on — it is never read from the frame) matches; a transplanted signature
     from another session fails here;
   - `bodyHash` recomputed from the received `body` bytes equals the hash in the signed message;
   - `method`/`path` in `stepUpMsg` equal the frame's;
   - `|now − ts| ≤ 60s`;
   - `crypto.verify('sha256', stepUpMsg, {key:<enrolled I_dev>, dsaEncoding:'ieee-p1363'}, sig)` is true
     for the device whose session this is.
   Any failure → `error ⇒ STEPUP_INVALID` and the action is **not** dispatched; missing `stepUp` on a
   high-risk route → `STEPUP_REQUIRED`.
5. The verified action is then dispatched into the normal route handler; the audit log records
   `{device, action, target, ts, result}` (design §4.6).

Read and low-risk controls carry **no** `stepUp`.

**Capability gate precedes step-up (LOCKED).** A HIGH route requires the session to **hold the
`highrisk` capability AND present a valid step-up** — and the capability is checked **first**. Sessions
carry a capability set fixed at handshake: a **cold** (Face-ID) PAIR/CONNECT session gets
`[read, lowrisk, mutate, highrisk]`; a **resumed** (ticket) session gets `[read, lowrisk]` only
(§2.4). The daemon dispatch therefore does, in order: classify tier (§8) → if HIGH and the session
lacks `highrisk` ⇒ **`CAP_DENIED`** (before any challenge/step-up) → else enforce step-up (§7.3 step 4).

This is the **true** enforcement of "a resumed session never reaches RCE", and it does **not** rest on
the device being unable to sign: the legitimate device holds the Secure-Enclave key in *every* session
type, so it could physically produce a valid step-up signature over a resumed session's `TH_with_kx`.
What blocks high-risk on a resumed session is the **missing capability**, not a missing signature. A
*stolen* ticket is doubly safe — it both lacks `highrisk` and cannot sign at all. To perform a
high-risk action the user must establish a fresh **cold** handshake (itself a Face-ID re-auth), which
mints a `highrisk`-capable session. `CAP_DENIED` (not `STEPUP_REQUIRED`) is the correct signal here, so
the client does not prompt for a biometric that would never be accepted.

## 8. Control ↔ REST mapping & capability tiers

dispatched into the existing `manager/routes/*` handler for that `method+path` via the design's
virtual-response shim. Each remotely-reachable route carries a **tier**:

- **READ** — no step-up. Pure state reads.
- **LOW** — no step-up. Non-RCE mutations a stolen read-only ticket is still *not* granted (they need a

- **REFUSED** — never dispatched remotely; returns `ROUTE_REFUSED`.

## 9. Interop test vectors & fixed-key fixture

Cross-platform byte-equality is verified at three levels. Where a value is a published standard KAT we
cite it; where it is composite, we fix the inputs and require the **first implementer to commit the
computed "golden" outputs**, which the other two MUST reproduce byte-for-byte (the design's
"fixed-key test fixture" requirement).

### 9.1 Primitive KATs (pull directly from the standard)

- **X25519** — RFC 7748 §6.1 (deterministic, confirms the ephemeral DH):
  ```
  aPriv = 77076d0a7318a57d3c16c17251b26645df4c2f87ebc0992ab177fba51db92c2a
  aPub  = 8520f0098930a754748b7ddcb43ef75a0dbf3a0d26381af4eba4a98eaa9b4e6a
  bPriv = 5dab087e624a8a4b79e17f8b83800ee66f3bb1292618b6fd1c2f8b27ff88e0eb
  bPub  = de9edb7d7b7dc1b4d35b61c2ece435373f8343c85b78674dadfc7e146f882b4f
  X25519(aPriv,bPub) = X25519(bPriv,aPub)
                     = 4a5d9d5ba4ce2de1728e3bf480350f25e07e21c947d19e3376f09b3c1e161742
  ```
  Each impl MUST reproduce the shared secret. (`crypto_kx` layers BLAKE2b on top — verified in §9.2.)
- **XChaCha20-Poly1305-IETF** — the CFRG draft (`draft-irtf-cfrg-xchacha`, Appendix A.3) AEAD vector;
  libsodium ships it as a self-test. Each impl MUST encrypt the draft's `(key, nonce24, aad,
  plaintext)` to the draft's exact `ciphertext‖tag` and decrypt back. (Reproduce from the cited draft
  rather than transcribed here, to avoid a transcription error becoming a spec error.)
- **BLAKE2b-256** — RFC 7693. Each impl MUST match libsodium's `crypto_generichash` self-test for both
  unkeyed and 32-byte-keyed modes.
- **P-256 ECDSA verify** — NIST CAVP `SigVer` P-256/SHA-256 vector (the *verify* direction; signing is
  randomized so it has no fixed output). Each impl's verifier MUST accept the NIST `(Qx,Qy,msg,r,s)`
  valid case and reject the tampered case, with `r‖s` decoded as the 64-byte raw form.

### 9.2 Composite golden fixture (commit-and-match)

**Ownership (PINNED):** the **daemon implementer (Node)** generates and commits this fixture to
`docs/vectors/ios-remote-v1/` **first**, before the relay and iOS clients wire their crypto. The
Node `sodium-native` + built-in `crypto` outputs are the reference bytes; the relay and iOS (Swift)
implementations MUST reproduce them byte-for-byte. The single **`{"t":"ka","ts":0}` data-frame
`ciphertext‖tag`** vector (below) is the **go/no-go interop gate**: an implementation that does not
reproduce those exact bytes is not interoperable and MUST NOT ship. All inputs are fixed so every
output except the two ECDSA signatures is deterministic.

```
FIXED INPUTS (hex):
  ePub_c / eSec_c : device ephemeral X25519  = RFC7748 "a" pair above
  ePub_s / eSec_s : Mac    ephemeral X25519  = RFC7748 "b" pair above
  clientNonce(16) : 00112233445566778899aabbccddeeff
  serverNonce(16) : ffeeddccbbaa99887766554433221100
  I_mac (P-256)   : <fixed test keypair, committed in the fixture dir>
  I_dev (P-256)   : <fixed test keypair, committed; NOTE: a fixture test key, NOT a Secure-Enclave key>
  ots(32)         : 0101...01 (32 bytes of 0x01)
  room(b64u22)    : "AAAAAAAAAAAAAAAAAAAAAA"   (b64u of 16 zero bytes; ASCII, roomLen=22=0x16)
  clientId(16)    : 000102030405060708090a0b0c0d0e0f
  -- room + clientId are REQUIRED inputs: they are part of the data-frame AEAD AAD (§1.5),
     so the ka ciphertext‖tag below is only deterministic once they are pinned. --

DERIVED — golden outputs to commit and cross-check:
  K_c2s, K_s2c                 (crypto_kx, role-fixed)
  TH2, TH3                     (BLAKE2b-256 transcript hashes)
  K_hs_s2c, K_hs_c2s           (KDF)
  K_c2s_final, K_s2c_final     (KDF over TH3)
  otsProof                     (keyed BLAKE2b over TH3)
  encS, encC bodies            (AEAD; deterministic GIVEN the signature bytes are pinned in the fixture)
  DATA-FRAME GOLDEN: plaintext = {"t":"ka","ts":0}  (exact 15 UTF-8 bytes, keys as written, no spaces)
     key   = K_c2s_final
     npub  = epoch(00) ‖ dir(00) ‖ seq(00000000 00000000 BE) ‖ 00×14   (24 bytes)
     AAD   = ver(01) ‖ epoch(00) ‖ dir(00) ‖ seq(8B=00..00) ‖ roomLen(16)
             ‖ room("AAAAAAAAAAAAAAAAAAAAAA" = 22 ASCII bytes) ‖ clientId(16B above)
     → ciphertext‖tag (fully deterministic given the above; THE go/no-go cross-platform gate).
       Commit the exact AAD byte string and the npub alongside the output so the other two impls
       diff against identical inputs, not just the final tag.
```

Because the two ECDSA signatures are randomized, the fixture **pins specific signature bytes** (a
valid `sig_s`/`sig_c` committed once) so the dependent `encS`/`encC` ciphertexts are also deterministic;
implementers verify by checking that (a) the pinned signatures *verify*, and (b) re-encrypting the same
sealed plaintext reproduces the committed ciphertext. The `ka` data-frame vector needs no signatures
and is the primary go/no-go interop gate.

| Side | Dependency | Version | Used for |
|---|---|---|---|
| Mac daemon (Node) | `sodium-native` | **≥ 5.0.0** | crypto_kx, crypto_generichash, xchacha20poly1305_ietf |

xchacha20poly1305_ietf_encrypt`) is mandatory (§1.5). CryptoKit is first-party (no SPM dep) and is the
**only** library that can touch the Secure Enclave.

## 11. End-to-end sequence (informative)

```
COLD PAIR (relay mode):
  Mac → relay : register{room, owner, allow:[hash(pairBearer)]}
  device scans QR → join{room, bearer=pairBearer}      relay: admit, assign clientId, tell Mac "joined"
  device ⇄ Mac : PAIR-1, PAIR-2, PAIR-3 (E2E, relay forwards opaque)   → device enrolled
  Mac → device : reply{ durableBearer, ticket{ticketId,PSK,idleExp,absExp} }   (sealed, traffic keys)

WARM RESUME (foreground, no Face ID):
  device → Mac : RES-1{ticketId, ePubC, binder}
  Mac → device : RES-2{ePubS, binder, encTicket=new}   → live again, read+lowrisk
  high-risk verb → control{path:/stepup/challenge} → challenge → SE sign (Face ID) → control{...,stepUp}
```

## 12. Unresolved interop questions

Each of these would benefit from a second crypto/security opinion (the peer-consult channel was
unavailable). None blocks a v1 build; each is flagged so it does not become a silent interop bug.

1. **Mac static key: P-256 vs Ed25519.** This spec makes the Mac identity P-256 for symmetry with the
   Enclave-forced device key. An alternative keeps the Mac key Ed25519 (libsodium `crypto_sign`,
   matching the original design wording) and has the device verify it via CryptoKit
   `Curve25519.Signing`. Both interoperate; P-256-both is fewer moving parts. **Decision taken:
   P-256 both.** Flagged only because it deviates from the design doc's "Ed25519" wording (the device
   side was *forced* to deviate regardless).
2. **AEAD: XChaCha (libsodium) vs IETF-ChaChaPoly (platform-native).** We kept the design's XChaCha
   (libsodium both ends). A documented alternative — IETF ChaCha20-Poly1305 with a 12-byte
   `epoch‖dir‖seq` nonce — would let iOS use **CryptoKit `ChaChaPoly`** and Node use built-in
   `crypto`, dropping the `sodium-native`/`swift-sodium` dependency entirely. It is equally safe given
   our deterministic counter-nonce. **Decision taken: XChaCha/libsodium** (honors the design; one
   library, identical bytes). Revisit if the native-addon build cost on the daemon/relay proves painful.
3. **Relay room-owner trust-on-first-use.** §5.2 has the relay pin the room-owner hash on first
   `register`. The exact provisioning (operator config vs TOFU) is a relay-deployment policy detail,
   not an E2E-crypto detail (relay compromise is already bounded to metadata+DoS by E2E). Needs a relay
   author's confirmation of the registration-secret distribution.
4. **`crypto_kx` BLAKE2b-512 internal KDF stability across libsodium majors.** Relied upon as the
   cross-platform DH-KDF. It has been stable for years, but the golden fixture (§9.2) is what actually
   guarantees the Node and iOS libsodium builds agree; pin both to a known libsodium version and
   verify the `K_c2s`/`K_s2c` vector in CI on both platforms.
5. **`/stepup/challenge` virtual route.** Introduced here as a daemon-side virtual route (not a worker
   route). Its handler must be added to the WsBridge dispatch, not `manager/routes/*`. Confirm placement
   with the Eos internals owner during implementation.

## 13. Pinned interop constants (single source of truth)

This section indexes the values that MUST be byte-identical across the relay (Node), daemon (Node), and
device (Swift). Where a value is fully specified elsewhere it is cross-referenced, not restated; the two
items most prone to silent drift — the **libsodium C-library version** and the **load-bearing ASCII
label strings** — are pinned in full here.

### 13.1 libsodium version (PINNED — one C library, both platforms)

- Both the Node binding (`sodium-native` ≥ 5.0.0) and the iOS binding (`swift-sodium` →
  `Clibsodium.xcframework`, pinned SPM commit) MUST resolve to **the same underlying libsodium C
  release: `libsodium 1.0.20`**. `crypto_kx`, `crypto_generichash` (BLAKE2b), and
  `crypto_aead_xchacha20poly1305_ietf` are deterministic and version-stable, but pinning the exact C
  release removes the last source of divergence. Each side records the resolved
  `sodium_version_string()` in CI.
- **CI gate (REQUIRED, both platforms):** the `crypto_kx` outputs **`K_c2s` and `K_s2c`** from the §9.2
  fixed ephemeral inputs MUST be recomputed and asserted equal to the committed golden values **on Node
  AND on iOS**. This is the concrete proof the two libsodium builds agree (it is what §12.4 ultimately
  rests on). A mismatch blocks the build.

### 13.2 Load-bearing ASCII label strings (PINNED, exact bytes, no NUL)

Every string below is an **ASCII byte sequence with no NUL terminator** (§0) that enters a crypto input
(a KDF `label`, a keyed-hash domain tag, or a signed-message prefix). A single differing byte breaks
interop silently — the AEAD/verify just fails. These exact bytes MUST be reflected in the §9.2 golden
fixture so any drift is caught by the data-frame gate. Implementers MUST embed these as literal ASCII,
not via locale/encoding-dependent helpers.

| String (exact) | Used in | Role |
|---|---|---|
| `eos/v1 hs s2c`           | §2.1 ②      | KDF label → handshake key `K_hs_s2c` |
| `eos/v1 hs c2s`           | §2.1 ⑤      | KDF label → handshake key `K_hs_c2s` |
| `eos/v1 pair server`      | §2.1 ③, §3.1 | Mac PAIR signature prefix |
| `eos/v1 pair client`      | §2.1 ⑦, §3.1 | Device PAIR signature prefix |
| `eos/v1 conn server`      | §2.2, §3.1   | Mac CONNECT signature prefix |
| `eos/v1 conn client`      | §2.2, §3.1   | Device CONNECT signature prefix |
| `eos/v1 data c2s`         | §2.5         | KDF label → traffic key `K_c2s_final` |
| `eos/v1 data s2c`         | §2.5         | KDF label → traffic key `K_s2c_final` |
| `eos/v1 resume binderC`   | §2.3         | Keyed-BLAKE2b (key=PSK) device resume binder |

### 13.3 Cross-referenced locked constants (specified in place)

- P-256 encodings: pubkey = 65-byte SEC1 `0x04‖X‖Y`, signature = 64-byte raw `r‖s` (IEEE-P1363, never
  DER); device identity is P-256 in the Secure Enclave (§1.2, §1.1).
- AEAD: XChaCha20-Poly1305-IETF, 24-byte structured `npub` via the explicit-nonce C call
  (`crypto_aead_xchacha20poly1305_ietf_encrypt`); swift-sodium's auto-nonce high-level API is
  **FORBIDDEN** (§1.5, §10).
- AEAD AAD binds the full outer header `ver‖epoch‖dir‖seq‖roomLen‖room‖clientId`, and the nonce's
  `epoch`/`dir`/`seq` MUST equal the AAD values — mismatch ⇒ `DECRYPT_FAIL` (§1.5, §4.1).
- `bodyHash` / `sessionTH` are computed over the **exact received bytes** with no re-serialization
  between hash and dispatch (§3.4, §3.2).
- Constant-time comparison for all secrets (bearers, owner secret, binders, `otsProof`); the one-time
  pairing `ots` is **burned on first accept** and the pairing endpoint is rate-limited / lockout-guarded
  (§2.1, §5.1).
- Golden fixture ownership + the `{"t":"ka","ts":0}` data-frame go/no-go gate (§9.2).
